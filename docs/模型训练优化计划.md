### T-01 前端训练作业提交表单字段名不匹配

**现状**

`TrainingJobListView.vue` 表单提交：
```json
{ "sceneId": "SCENE-SINTER-FIRE-001", "datasetCode": "DS-SINTER-001", "epochs": 100 }
```

后端 `TrainJobRequest` 期望（通过 `@JsonProperty`）：
```json
{ "plugin_id": "ATOM-DETECT-YOLO-V1", "dataset_version_id": "DS-SINTER-001-v1.0", "train_config_json": {"epochs": 10}, "gpu_count": 1 }
```

**问题**：字段名、字段语义完全不一致，提交后 `plugin_id` / `dataset_version_id` 均为 null，`@NotBlank` 校验失败，返回 400。

**修复方案**

修改 `TrainingJobListView.vue` 提交表单：
- 将 `sceneId`（场景 ID）改为 `pluginId`（算法插件 ID），从插件列表下拉选择
- 将 `datasetCode`（数据集编号）改为 `datasetVersionId`（数据集版本 ID），从数据集版本列表选择
- 将 `epochs: number` 改为嵌套在 `trainConfigJson.epochs` 中
- 新增 `gpuCount` 字段（默认 1，CPU 模式时无效但后端需要）

修改后发送格式：
```json
{
  "plugin_id": "ATOM-DETECT-YOLO-V1",
  "dataset_version_id": "DS-SINTER-001-v1.0",
  "gpu_count": 1,
  "train_config_json": { "epochs": 10, "img_size": 640, "batch_size": 8 }
}
```

**负责人**：FE-1 + BE-2 | **工时**：0.5 天

### T-02 训练启动逻辑未实现

**现状**

`TrainingController.submitJob()` 仅在 `train_job` 表插入一条 PENDING 记录。Sprint 开发计划 S1-17 标注为 ✅ 但代码内有注释：
```java
// 实际训练 K8s Job 创建逻辑（Sprint 3 完善）
return ApiResponse.ok(Map.of("message", "重训练任务已提交", ...));
```

**修复方案（两阶段）**

#### 阶段一：Sprint 3 测试环境 — `docker run` 方案（无需 K8s）

测试环境为 3 台物理服务器，无 K8s 集群，用 `ProcessBuilder` 直接在 Server 1 本地执行 `docker run` 启动训练容器：

```java
// drift-monitor-service: TrainingController.submitJob() 末尾追加
@Component
public class DockerTrainJobLauncher {

    @Value("${tianjing.train.docker-image:tianjing/yolo-trainer:cpu-latest}")
    private String trainerImage;

    public void launch(TrainJob job, String trainConfigJson) {
        // 构建 docker run 命令（非阻塞，训练完成后由训练脚本回调更新状态）
        List<String> cmd = new ArrayList<>(List.of(
            "docker", "run", "--rm", "-d",
            "--name", "train-" + job.getJobId().toLowerCase(),
            "-e", "JOB_ID=" + job.getJobId(),
            "-e", "PLUGIN_ID=" + job.getPluginId(),
            "-e", "DATASET_VERSION_ID=" + job.getDatasetVersionId(),
            "-e", "TRAIN_CONFIG_JSON=" + trainConfigJson,
            "-e", "MINIO_ENDPOINT=" + minioEndpoint,
            "-e", "MLFLOW_TRACKING_URI=" + mlflowUri,
            "-e", "PLATFORM_CALLBACK_URL=http://host-gateway:8089",
            "--add-host=host-gateway:host-gateway",
            trainerImage
        ));
        new ProcessBuilder(cmd).start();
        log.info("启动训练容器 job_id={}", job.getJobId());
    }
}
```

**配置项**（`application.yml` 新增）：
```yaml
tianjing:
  train:
    docker-image: tianjing/yolo-trainer:cpu-latest
    launcher: docker   # docker | k8s，Sprint 5 切换为 k8s
```

#### 阶段二：Sprint 5 正式环境 — K8s Job 方案

GPU 服务器就位、K8s 集群建立后，将 `DockerTrainJobLauncher` 替换为 `KubernetesTrainJobLauncher`（使用 `io.fabric8:kubernetes-client`），通过配置 `tianjing.train.launcher=k8s` 切换，无需修改 Controller 层代码。

**负责人**：BE-2 | **工时**：1.5 天（docker run 方案，比 K8s 方案减少 0.5 天）

### T-03 训练脚本不存在

**现状**

`training/` 目录全部为空（`.gitkeep`），没有任何可执行的训练代码。

**修复方案**

在 `training/train_jobs/` 中创建 CPU 模式训练脚本 `train_yolo_cpu.py`，功能覆盖：

```
① 从环境变量读取参数（JOB_ID, PLUGIN_ID, DATASET_VERSION_ID, EPOCHS 等）
② 从 MinIO 下载数据集（tianjing-datasets/{factory}/{dataset_code}/）
③ 解析标注文件（Label Studio COCO 格式 → YOLO 格式转换）
④ 加载 YOLOv8n 预训练权重（ultralytics）
⑤ 执行训练（CPU 模式，epochs=10，用于流程验证）
⑥ 训练过程中每 epoch 回调更新 train_job 状态
⑦ 训练完成后：
   - 导出 ONNX 格式模型
   - 向 MLflow 记录指标和注册模型
   - 调用平台回调接口更新 train_job 状态为 COMPLETED + 写入 mlflow_run_id + best_map50
```

同时在 `training/train_jobs/` 中创建 `Dockerfile`，基于 `python:3.11-slim`，安装 `ultralytics`、`mlflow`、`boto3`（MinIO）、`requests`。

**V2.0 精度要求**：CPU 训练仅验证流程，不要求精度达标（Sprint 3 精度门槛宽松）。

**负责人**：Algo-2 + BE-2 | **工时**：3 天

### T-04 训练状态无回写机制

**现状**

K8s Job 运行完成后，`train_job` 表的 `status` 字段无人更新，`mlflow_run_id`、`best_map50`、`finished_at` 均为空。

**修复方案**

在 `drift-monitor-service` 中新增训练回调接口：

```
POST /internal/training/jobs/{job_id}/callback
Body: { status, mlflow_run_id, best_map50, best_map50_95, error_msg }
```

训练脚本在以下时机调用此接口：
- 训练开始：`status=RUNNING`
- 每 epoch 完成：更新 `best_map50`（用于前端进度展示）
- 训练完成：`status=COMPLETED` + `mlflow_run_id` + 指标
- 训练失败：`status=FAILED` + `error_msg`

同时在 `drift-monitor-service` 中新增超时扫描任务，防止训练容器异常崩溃时状态卡死：

```java
// 每 30 秒扫描一次 RUNNING 状态超 2 小时未更新的 job，标记为 FAILED
// docker run 和 K8s Job 两种模式均适用
@Scheduled(fixedDelay = 30_000)
public void scanStaleJobs() { ... }
```

**负责人**：BE-2 | **工时**：1.5 天

### T-05 训练完成后未自动注册模型版本

**现状**

训练完成后，需要人工到"模型版本管理"页面手动填表提交。无任何自动化衔接。

**修复方案**

训练脚本在回调 `COMPLETED` 时，同时调用 `alarm-rule-service` 的 `POST /api/v1/models` 接口，自动创建 `model_version` 记录（状态 `STAGING`）：

```python
# 训练完成后自动注册
payload = {
    "plugin_id": os.environ["PLUGIN_ID"],
    "version": f"auto-{job_id[-8:]}-{datetime.now().strftime('%Y%m%d')}",
    "mlflow_run_id": run.info.run_id,
    "model_artifact_url": f"models:/{plugin_id}/{mlflow_version}",
    "training_job_id": os.environ["JOB_ID"]
}
requests.post(f"{PLATFORM_API_URL}/api/v1/models", json=payload,
              headers={"Authorization": f"Bearer {INTERNAL_TOKEN}"})
```

注册成功后，前端训练作业详情页展示"已自动注册为模型版本 MV-XXXXXXXX，点击查看"。

**负责人**：Algo-2 + BE-2 | **工时**：1 天（与 T-03 合并）

### T-08 `drift-monitor-service` 缺少 `MetaObjectHandler`

**现状**

`TrainJob` 实体的 `createdAt` 字段标注了 `@FieldFill.INSERT`，但 `drift-monitor-service` 的 `MybatisPlusConfig.java` 中没有 `MetaObjectHandler` Bean，导致插入训练作业时 `created_at` 为 null 报 500。

**修复方案**

参考 `calibration-service` 已有修复，在 `drift-monitor-service` 的 `MybatisPlusConfig.java` 中补充：

```java
@Bean
public MetaObjectHandler metaObjectHandler() {
    return new MetaObjectHandler() {
        @Override
        public void insertFill(MetaObject metaObject) {
            this.strictInsertFill(metaObject, "createdAt",
                OffsetDateTime.class, OffsetDateTime.now());
        }
        @Override
        public void updateFill(MetaObject metaObject) { }
    };
}
```

**负责人**：BE-2 | **工时**：0.5 天

modelApi.submit({

```

  录像回放 → Kafka → 云端推理 → 告警 → Sandbox 拦截

  训练链路产出模型 → 进入推理链路验证效果
```

## 五、Sprint 5（GPU 到货后）训练升级计划

Sprint 3 的 CPU 训练仅验证链路，精度不达标。GPU 到货后的 Sprint 5 需在以下方面升级：

| 升级点 | Sprint 3（CPU 模式） | Sprint 5（GPU 模式） |
|---|---|---|
| 训练容器镜像 | `yolo-trainer:cpu-latest`（ultralytics CPU） | `yolo-trainer:gpu-trt`（PyTorch + TensorRT） |
| 训练规模 | epochs=10，用于验证流程 | epochs≥100，精度达标 mAP≥0.85 |
| 模型格式 | ONNX（CPU 推理） | TensorRT FP16 Engine（GPU 推理） |
| K8s 调度 | `training` 命名空间 CPU 节点 | GPU 节点 + `gpu: nvidia.com/gpu: 1` |
| 精度门槛 | 宽松（验证链路即可） | `best_map50 ≥ 0.85`，低于此值作业标记 FAILED |
| Sandbox 验证 | 跳过（Sprint 3 用配置绕过时长限制） | ≥48h Sandbox 精度对比，高于生产 2% 才转正 |
| TensorRT 打包 | 不需要 | `training/export/` 脚本自动完成 FP16 量化导出 |

Sprint 5 S5-05 具体场景训练顺序（按 P1 优先级）：
1. 铸坯表面缺陷（炼钢厂，Algo-1）
2. 链篦机侧板跑偏（球团厂，Algo-2）
3. 烧结机壁条检测（烧结厂，Algo-2，Sprint 3 CPU 版升级）
4. 型钢冷床质量（型钢厂，Algo-1）
5. 带钢端面质检（带钢厂，Algo-3）

## 六、风险与应对

| 风险 | 可能性 | 应对方案 |
|---|---|---|
| Server 1 内存不足，CPU 训练容器 OOM | 中 | 降低 `batch_size` 至 4；`epochs=5` 仅验证流程；或在 Server 2/3 之一跑训练容器 |
| MLflow Tracking Server 未部署或不可达 | 中 | S3-13 可降级：训练完成后人工在 MLflow UI 注册，自动注册列入 Sprint 4 |
| `docker run` 中 `host-gateway` 无法解析（网络配置问题） | 中 | 改为直接写 Server 1 的内网 IP：`-e PLATFORM_CALLBACK_URL=http://10.30.10.49:8089` |
| 烧结看火标注数据量不足（< 500 帧），CPU 训练损失不收敛 | 低 | Sprint 3 精度不做要求；使用 YOLOv8n 预训练权重 fine-tune，少量数据也能运行 |
| `tianjing_train` 库在开发环境不可连接 | 低 | 补充种子数据（T-07）后需重新验证数据源切换配置 |
| Sprint 5 K8s Job 替换时接口不兼容 | 低 | T-02 已通过配置项 `tianjing.train.launcher=docker/k8s` 隔离，替换只改 Bean，不改 Controller |
