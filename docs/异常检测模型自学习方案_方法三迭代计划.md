# 异常检测模型自学习方案 — 方法三：先跑起来，再迭代标注

> **状态：已评审，同意实施**
> **日期**：2026-04-18　**评审通过日期**：2026-04-21
> **作者**：天柱·天镜项目组
> **背景**：仿真任务中上传录像时已支持正常/异常标签选项，基于此探索通过图像级标签自动训练推理模型的技术路线，避免前期大量人工框选标注的成本。

---

## 一、核心思路

传统路线要求对每一帧画精确 Bounding Box（边界框）才能训练检测模型，成本高、周期长。
本方案采用**图像级弱监督标签**驱动，分阶段迭代，以最低初始标注成本快速上线，再逐步精确化。

```
图像级标签（正常/异常/异常类型）
        ↓
  分类模型 / 异常检测模型（ONNX）
        ↓
  接入现有推理链路
        ↓
  收集误报/漏报 → 补充 BBox 标注 → 升级 YOLO 检测模型
```

---

## 二、两条技术路线对比

| 维度 | 图像级标签（本方案） | 传统 BBox 标注训练 |
|---|---|---|
| 标注成本 | 极低（只需正常/异常分类） | 高（每帧画框 + 类别） |
| 输出能力 | "这帧有问题" | "第 (x1,y1)-(x2,y2) 处有篦条断裂" |
| 定位精度 | 无或粗略热力图 | 精确像素级 |
| 训练数据量 | 少量异常即可起步 | 需足量各类缺陷样本 |
| 适合阶段 | 冷启动、快速验证 | 精确化阶段 |

---

## 三、整体路线图

```
阶段一（约 2 周）    阶段二（约 2 周）    阶段三（约 1 周）    阶段四（持续迭代）
数据采集与标注 ───→ 模型训练与导出 ───→ 接入推理链路 ───→ 迭代标注升级
上传 + 元数据入库    训练 + ONNX 导出    报警联动验收       精确化 → YOLO
```

## 五、阶段二：模型训练与导出（第 3–4 周）

**目标**：训练出可用的 ONNX 分类/异常检测模型

### 5.1 模型选型

| 适用情形 | 推荐方法 | 说明 |
|---|---|---|
| 只有正常帧、异常罕见 | **PatchCore**（anomalib） | 零异常标注，只用正常帧训练 |
| 有正常 + 多类异常帧（通用缺陷场景） | **EfficientNet-B2 分类器** | 图像级标签直接训，参数少、精度高、ONNX 兼容 |
| 火焰 / 热工强度场景 | **EfficientNet-B2 三分类** | 输出过弱 / 正常 / 过强，覆盖双向偏差 |

> **第一版本按场景分别选型**：
> - 通用缺陷场景（篦条缺损、铸坯裂纹等）→ **EfficientNet-B2**，plugin_id: `CLASSIFY-ANOMALY-V1`
> - 火焰/热工强度场景（烧结机火焰等）→ **EfficientNet-B2 三分类**，plugin_id: `CLASSIFY-FLAME-V1`
>
> 同步保留 **PatchCore** 作为兜底，检测未见过的新型异常。
>
> ⚠️ 注：原方案选用 ResNet-50，经评审升级为 EfficientNet-B2（精度+4%、推理速度+40%、参数量-64%，ONNX 导出完全兼容）。

### 5.2 火焰场景三分类设计（评审新增）

适用场景：`SCENE-SINTER-FIRE-*` 等火焰/热工强度监测场景。

```python
import timm

# 模型定义
model = timm.create_model("efficientnet_b2", pretrained=True, num_classes=3)
# 输出顺序：[过弱, 正常, 过强]，具有语义顺序

# 损失函数：加权交叉熵（过弱/过强样本通常较少）
criterion = nn.CrossEntropyLoss(weight=torch.tensor([1.5, 1.0, 1.5]))

# 推理输出映射
FLAME_LABELS = ["过弱", "正常", "过强"]
pred_idx = outputs.softmax(-1).argmax(-1)
label_name = FLAME_LABELS[pred_idx]
anomaly_score = 1.0 - outputs.softmax(-1)[1]  # 非"正常"的综合置信度
```

训练依赖新增：`timm>=0.9.0`（需加入 `Dockerfile.train`）

### 5.3 训练脚本

新增：`training/train_jobs/anomaly_classify/train.py`

```
核心流程：
1. 从 MinIO 读取 normal/ 和 abnormal/{type}/ 帧
2. EfficientNet-B2 fine-tune（冻结前 6 层，只训后续层 + FC 任务头）
3. Grad-CAM 对验证集误报/漏报帧生成热力图 → 存 MLflow Artifacts（gradcam/）
4. MLflow 记录 accuracy / F1 / confusion matrix + Grad-CAM 可视化报告
5. 导出 ONNX（opset 17，input shape: [B, 3, 640, 640]）
6. 推送模型至 Harbor: tianjing-models-staging
```

**Grad-CAM 热力图生成（评审新增）**：

```python
from pytorch_grad_cam import GradCAM
from pytorch_grad_cam.utils.image import show_cam_on_image

# 对最后一个卷积块生成热力图
cam = GradCAM(model=model, target_layers=[model.blocks[-1]])

# 对验证集中的误报/漏报帧批量生成，存入 MLflow
for frame_path, img_tensor, label in misclassified_frames:
    grayscale_cam = cam(input_tensor=img_tensor.unsqueeze(0))
    visualization = show_cam_on_image(raw_img / 255.0, grayscale_cam[0])
    mlflow.log_image(visualization, f"gradcam/{label}/{Path(frame_path).name}")
```

> **注意**：Grad-CAM 仅在训练/标注阶段使用 PyTorch checkpoint，**不集成到生产 ONNX 推理链路**。
> ONNX 格式不支持梯度计算，生产推理镜像无需变动。

训练依赖新增：`grad-cam==1.4.8`（需加入 `Dockerfile.train`，生产推理镜像不需要）

### 5.4 推理输出适配

分类模型无 BBox，需适配现有 `InferOutput` schema（下游 alarm-judge-service 无需改动）：

```python
# 分类结果 → InferOutput detections[] 格式
if anomaly_score > threshold:
    detections = [Detection(
        class_name=anomaly_type_name,   # 如 "篦条缺损" 或 "过强"
        confidence=anomaly_score,
        bbox={"x1": 0, "y1": 0,         # 全图 BBox，表示"整帧异常"
              "x2": W,  "y2": H},
    )]
else:
    detections = []
```

新增 plugin_id：`CLASSIFY-ANOMALY-V1`（通用缺陷）、`CLASSIFY-FLAME-V1`（火焰三分类）

**阶段二交付物**：
- MLflow 中注册的 ONNX 模型（按场景分别注册）
- Grad-CAM 热力图验证报告（存于 MLflow Artifacts）
- 精度报告（F1 ≥ 0.85 方可进入阶段三）

### 6.2 验收标准

```
✅ 播放"异常"录像 → 推理 confidence ≥ 0.8 且连续 2 帧 → 触发 WARNING 级报警

```

## 七、阶段四：迭代标注，逐步精确化（持续）

上线后收集误报/漏报数据，逐步升级模型精度与定位能力：

```
第一轮（上线后 2 周）
├── 收集误报帧（模型说异常但实际正常）
├── 收集漏报帧（模型未报但人工复核确认异常）
├── 标注审核页面展示 Grad-CAM 热力图 → 辅助标注人员定位异常区域
└── 人工对这些帧补画 BBox（参考热力图建议位置）→ 加入训练集

第二轮（上线后 1 个月）
├── 每类异常积累 ≥ 500 帧 + BBox
├── 基于 MMDetection 训练 YOLOv9 检测头
└── 替换 ONNX 分类模型 → YOLO 检测模型，精确定位生效

长期
└── drift-monitor-service 自动监测精度下降 → 触发重训练流水线
```

### 7.1 Grad-CAM 标注辅助工具（评审新增）

在 `AnnotationReviewView.vue` 标注审核页面增加热力图参考功能：

```
标注人员查看待回标帧
        ↓
点击"热力图参考"按钮
        ↓
后端调用 PyTorch checkpoint（非 ONNX）运行 Grad-CAM
        ↓
返回热力图叠加图像（红色高亮区域 = 模型认为异常的位置）
        ↓
标注人员参考高亮区域画 BBox，提升标注效率和准确性
```

- 接口：`POST /api/v1/simulations/{task_id}/review/frames/{frame_id}/gradcam`
- 响应：热力图叠加图的 Base64 或 MinIO 临时 URL
- 调用的是阶段二训练时保存的 `.pt` checkpoint，与生产 ONNX 模型完全隔离

## 八、关键风险与应对

| 风险 | 应对措施 |
|---|---|
| 异常样本太少（< 100 帧/类） | 改用 PatchCore，只需正常帧即可训练 |
| 误报率高导致运维投诉 | 告警连续帧数由 2 调至 3，置信度阈值提高至 0.88 |
| 全图 BBox 导致第四阶段回标浪费 | Grad-CAM 热力图辅助标注，缩小人工画框范围，降低回标成本 |
| 模型只识别训练过的异常类型 | 并行部署 PatchCore 作兜底，检测未见类型 |
| 训练数据分布不均（正常帧 >> 异常帧） | 训练时对异常类别进行过采样（WeightedRandomSampler） |
| Grad-CAM 热力图误导标注人员（关注区域不准） | 热力图仅作参考，最终 BBox 由人工确认；训练报告中展示热力图质量，F1 < 0.85 时不开放热力图辅助 |
| 火焰三分类样本中"过弱"极少（安全原因难采集） | 初期可用视频调色/亮度变换做数据增强模拟过弱样本 |
