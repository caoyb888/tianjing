# 仿真任务一键生成训练集方案

> **版本**：V1.0 · **日期**：2026-04-05
> **背景**：当前训练作业提交需要手动抽帧、标注、打包、上传 MinIO，步骤繁多。
> 本方案复用已有的仿真任务（`history-replay-service`）基础设施，将"视频→推理→标注→数据集"全链路集成进实验室模块，实现一键生成训练集。

---

## 一、现状与复用基础

### 已有能力（可直接复用）

| 已有组件 | 可复用内容 |
|---|---|
| `SimulationController` / `SimulationService` | 视频上传、任务管理、MinIO 读写 |
| `simulation_task.result_json` | JSONB 字段，已设计用于存储帧级推理结果 |
| `simulation_task.algo_config_json` | 记录使用的插件 ID 和推理参数 |
| `tianjing-sim-temp` bucket | 视频文件已上传至此 |
| `tianjing-datasets` bucket | 目标数据集存储桶，格式已约定 |
| `dataset` / `dataset_version` 表 | `tianjing_train` 库，结构已完备 |
| Sandbox 推理拦截器 | 推理结果带 `is_sandbox=true`，不触发生产告警 |

### 当前缺口（需要补充）

| 缺口 | 说明 |
|---|---|
| 仿真执行引擎 | 任务创建后状态停在 `PENDING`，没有实际抽帧和推理 |
| 推理结果→COCO 转换 | 需要将 `result_json` 中的检测框转为 COCO 标注格式 |
| 数据集打包导出 | 需要将帧图像 + COCO 文件打包上传 `tianjing-datasets` |
| 前端导出入口 | `SimulationDetailView.vue` 完成后无"生成训练集"按钮 |

## 二、完整流程设计

```
① 上传视频
   管理后台 → 仿真任务 → 新建仿真任务
   选择场景 + 插件 → 上传 MP4
   ↓ 视频存入 tianjing-sim-temp/simulation/{sceneId}/{uuid}/video.mp4

② 仿真推理（抽帧 + 自动标注）
   history-replay-service 执行引擎：
   ├─ ffmpeg 按配置抽帧（默认 1fps，可调）
   ├─ 帧图像存入 tianjing-sim-temp/simulation/{taskId}/frames/
   ├─ 逐帧调用 sandbox-infer-service（gRPC，is_sandbox=true）
   ├─ 推理结果写入 simulation_task.result_json
   └─ 进度回写 simulation_task.progress（前端 10s 轮询）

③ 仿真完成
   状态 → COMPLETED
   详情页显示推理结果摘要（帧数、检测框数、高置信度比例）

④ 一键导出训练集（新功能）
   用户点击「导出为训练数据集」→ 填写对话框：
   ├─ 目标数据集版本 ID（新建或选择已有未冻结版本）
   ├─ 置信度阈值（默认 0.7，低于此值的检测框不纳入标注）
   └─ 是否包含负样本帧（无检测结果的帧，默认包含）
   ↓
   history-replay-service 执行：
   ├─ 读取 result_json 中的推理结果
   ├─ 过滤低置信度框（< threshold）
   ├─ 转换为 COCO JSON 格式
   ├─ 打包 images/ + annotations.json → dataset.zip
   ├─ 上传到 tianjing-datasets/{datasetVersionId}/dataset.zip
   ├─ 更新 tianjing_train.dataset_version.sample_count
   └─ 回写 simulation_task.dataset_version_id（记录溯源）

⑤ 直接提交训练作业
   导出成功后，详情页出现「提交训练作业」快捷按钮
   ↓ 跳转到 /training/jobs，数据集版本预填
```

## 三、新增接口设计

### 3.1 `history-replay-service` 新增 2 个接口

#### POST `/simulations/{task_id}/export-dataset` — 触发导出

```json
// 请求体
{
  "dataset_version_id": "DV-MATERIAL-LEVEL-V1-2",   // 新版本 ID，不存在时自动创建
  "dataset_code":       "DS-MATERIAL-LEVEL-001",     // 所属数据集
  "conf_threshold":     0.70,                         // 置信度过滤阈值
  "include_negatives":  true                          // 是否包含负样本帧
}

// 响应
{
  "code": 0,
  "data": {
    "export_task_id": "EXP-XXXXXXXX",  // 异步导出任务 ID
    "dataset_version_id": "DV-MATERIAL-LEVEL-V1-2"
  }
}
```

#### GET `/simulations/{task_id}/export-status` — 查询导出进度

```json
{
  "code": 0,
  "data": {
    "status": "RUNNING",         // PENDING / RUNNING / COMPLETED / FAILED
    "progress": 65,              // 百分比
    "total_frames": 300,
    "exported_frames": 195,
    "annotation_count": 412,     // 生成的标注框数
    "dataset_version_id": "DV-MATERIAL-LEVEL-V1-2",
    "minio_path": "tianjing-datasets/DV-MATERIAL-LEVEL-V1-2/dataset.zip"
  }
}
```

### 3.2 Nginx 网关 — 无需新增路由

`/api/v1/simulations` 已路由至 `history-replay-service`，子路径自动覆盖。

### 5.2 推理结果 → COCO 转换

```java
// DatasetExportService.java（新增）
public class DatasetExportService {

    public void exportDataset(SimulationTask task, ExportRequest req) {
        // 1. 读取 result_json（已存储的推理结果）
        List<FrameResult> results = parseResultJson(task.getResultJson());

        // 2. 过滤低置信度检测框
        results = filter(results, req.confThreshold(), req.includeNegatives());

        // 3. 构建 COCO 结构
        CocoDataset coco = buildCoco(results);  // images[] + annotations[] + categories[]

        // 4. 下载帧图像 + 打包 zip
        Path zipPath = packDataset(task.getTaskId(), results, coco);

        // 5. 上传到 tianjing-datasets/{datasetVersionId}/dataset.zip
        minioClient.uploadObject(...);

        // 6. 更新 dataset_version.sample_count 和 simulation_task.export_status
        updateRecords(task, req.datasetVersionId(), results.size());
    }
}
```

## 七、实施分阶段计划

### Phase 1 — 导出引擎（1.5 天，可独立测试）
> 前提：仿真执行引擎暂用 mock result_json（手动构造一批检测结果），优先验证导出链路

- `DatasetExportService`：result_json → COCO → zip → MinIO
- `POST /simulations/{task_id}/export-dataset` 接口
- `GET /simulations/{task_id}/export-status` 接口
- V5 Migration：simulation_task 新增 2 列
- 前端：SimulationDetailView 导出区域 + 进度展示

**验收**：能从一条有 result_json 的 simulation_task 记录导出 dataset.zip 到 MinIO，tianjing_train.dataset_version 记录 sample_count 正确更新。

### Phase 2 — 仿真执行引擎（2 天）
> 打通视频 → 抽帧 → 推理 → result_json 全链路

- `SimulationExecutor`：ffmpeg 抽帧 + 虚拟线程并发推理
- 对接 `sandbox-infer-service`（gRPC 或 REST）
- 进度回写（每 10 帧更新一次 progress）
- 前端：新建对话框补充插件选择 + 抽帧频率

**验收**：上传烧结看火录像 → 仿真任务自动运行 → COMPLETED → result_json 有数据。

### Phase 3 — 全链路打通（0.5 天）
> Phase 1 + Phase 2 联调

- 端到端：上传录像 → 仿真推理 → 一键导出 → 直接跳转提交训练作业

## 八、与现有模块的边界约束

| 约束 | 说明 |
|---|---|
| Sandbox 隔离 | 推理调用必须带 `is_sandbox=true`，推理结果不触发生产告警 |
| 数据集存储桶 | 导出写入 `tianjing-datasets`（非 `tianjing-sim-temp`），遵守 MinIO 存储规范 |
| dataset_version 写入 | 只写 `tianjing_train` 库，不跨库访问生产库 |
| 帧图像保留 | sim-temp 中的帧图像 72h 后自动清理，dataset.zip 长期保留 |
| 导出不触发训练 | 导出仅生成数据集，不自动提交训练作业（由用户点击决定） |

## 九、Sprint 3 快速验收路径

完成 Phase 1 后，即可进行 S3-13 验证，无需等待 Phase 2：

```
1. 手动向数据库插入一条带 result_json 的 SIM 记录（可用现有种子数据）
2. 在仿真详情页点击「导出为训练数据集」
3. 确认 MinIO tianjing-datasets/DV-MATERIAL-LEVEL-V1-1/dataset.zip 存在
4. 跳转提交训练作业 → PENDING → docker run 训练 → COMPLETED
```

Phase 2（真实推理链路）可在 Sprint 4 补齐，Sprint 3 不阻塞 S3-13 验收。
