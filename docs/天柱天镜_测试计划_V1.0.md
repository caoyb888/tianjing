## 1. 测试目标与范围

### 1.1 测试目标

| 目标 | 具体描述 |
|---|---|
| **功能正确性** | 16 个检测场景全链路端到端贯通，告警判定逻辑准确，业务规则严格执行 |
| **Sandbox 安全隔离** | 实验室推理结果绝不触发 MQTT 推送、工单生成或 SCADA 联动（P0 红线） |
| **推理性能** | GPU 服务器单帧 P99 ≤ 50ms，16 并发场景 P99 ≤ 100ms |
| **系统稳定性** | 5 路场景同时回放 8 小时无服务崩溃，Kafka 消费无积压 |
| **安全合规** | ZAP/SQLMap 扫描无高危漏洞，JWT 鉴权、权限隔离、数据脱敏全部通过 |
| **业务约束验证** | 四眼原则、乐观锁、连续帧确认、时窗归并等核心约束全部有测试覆盖 |

### 1.2 测试范围

#### 纳入测试

- **后端微服务**（18 个 Spring Boot 服务，端口 8081–8101）
- **推理服务**（Python FastAPI，cloud-inference-proxy :8092、recording-replay :8091）
- **前端管理后台**（admin-web，Vue 3 + Element Plus，28 个视图页面）
- **中间件集成**（Kafka 消息流、PostgreSQL 持久化、TDengine 时序写入、MinIO 文件存取、Redis 缓存）
- **算法插件**（7 个原子算法 + 8 个场景任务头，Image In → JSON Out 接口）
- **工业互联网对接**（EMQX MQTT，Topic：`iiot/tianjing/alarm/{factory}`）
- **Sandbox 全链路**（T 型分流 → Sandbox 推理 → 拦截器 → 对比报告 → 转正门禁）
- **CI/CD 流水线**（GitHub Actions，三个门禁：代码质量、Sandbox 验证、生产部署）

#### 暂不纳入（待 Sprint 5+）

- 实时 RTSP 流接入（Sprint 5 GPU 到货后）
- 边缘端 Jetson / 海思 3559 推理（Sprint 5）
- 飞剪场景端到端 < 10ms 测试（需工业现场环境）
- 可视化大屏 monitor-screen（Sprint 6）

## 2. 测试用例设计策略

### 2.1 设计方法论

本项目采用 **分层测试策略**，从底层单元到顶层系统逐层验证，结合工业场景的特殊约束（Sandbox 隔离、四眼原则、数据安全）增设专项测试层。

```
┌─────────────────────────────────────────────┐
│         E2E 场景测试（16 个检测场景）          │  ← 烧结/炼钢/型钢/带钢/球团
├─────────────────────────────────────────────┤
│         系统集成测试（跨服务链路）             │  ← 推理链路 / 告警链路 / 转正链路
├─────────────────────────────────────────────┤
│    Sandbox 隔离专项测试 / 安全测试            │  ← P0 红线防护验证
├─────────────────────────────────────────────┤
│         API 接口测试（58 个端点）              │  ← 契约测试 + 异常路径
├─────────────────────────────────────────────┤
│         单元测试（算法插件 / 核心业务逻辑）    │  ← ≥ 90% 覆盖率
└─────────────────────────────────────────────┘
```

### 2.2 单元测试用例设计

#### 2.2.1 算法插件（覆盖率 ≥ 90%）

每个算法插件（`ATOM-DETECT-YOLO-V1` 等）必须包含以下用例分组：

| 用例分组 | 测试场景 | 验收标准 |
|---|---|---|
| **正常路径** | 标准工业图像（1920×1080，BGR 格式） | 输出格式严格符合 `InferOutput` Schema |
| **噪声图像** | 模糊图（拉普拉斯方差 < 50） | 不崩溃，`detections` 可为空列表 |
| **低亮度** | 黑屏/过暗（平均亮度 < 30） | 不崩溃，`inference_time_ms > 0` |
| **过曝** | 强光过曝（平均亮度 > 240） | 不崩溃，confidence 不超出 [0,1] |
| **遮挡** | ROI 30% 以上被遮挡 | 检测不崩溃，结果在 ROI 范围内 |
| **边界 ROI** | ROI 宽高等于图像尺寸边界值 | 不越界，不抛出 IndexError |
| **Sandbox 标识** | `is_sandbox=true` 的输入 | `InferOutput.is_sandbox` 原样透传，推理逻辑不变 |
| **耗时记录** | 任意正常图像 | `inference_time_ms > 0` 且写入输出 |
| **纯函数性** | 同一输入调用两次 | 输出完全一致（无状态副作用） |
| **元信息完整性** | `get_metadata()` | 包含 plugin_id、version、accuracy_metrics |

```python
# 用例模板示例
class TestAtomDetectYolo:

    def test_sandbox_flag_not_modified(self):
        """is_sandbox 标识在推理过程中不得被修改（P0 约束）"""
        inp = InferInput(..., is_sandbox=True)
        out = plugin.infer(inp)
        assert out.is_sandbox == True  # 原样透传

    def test_roi_detection_boundary(self):
        """检测结果必须在 ROI 范围内"""
        roi = ROI(x=100, y=100, w=400, h=300)
        out = plugin.infer(InferInput(image=img, roi=roi, ...))
        for det in out.detections:
            assert det.bbox['x1'] >= roi.x
            assert det.bbox['y1'] >= roi.y
            assert det.bbox['x2'] <= roi.x + roi.w
            assert det.bbox['y2'] <= roi.y + roi.h
```

#### 2.2.2 告警判定服务（覆盖率 ≥ 95%，Sandbox 拦截 100%）

| 用例 | 输入条件 | 期望行为 |
|---|---|---|
| **Sandbox 告警绝对拦截** | `is_sandbox=true`，`has_anomaly=true`，任意级别 | `notification_service.send` 未被调用；`sandbox_db.write` 已调用 |
| **生产 CRITICAL 告警推送** | `is_sandbox=false`，置信度 ≥ 0.9，连续 3 帧 | MQTT + 企业微信 + 工单三通道全部触发 |
| **生产 WARNING 告警推送** | `is_sandbox=false`，置信度 ≥ 0.8，连续 2 帧 | WebSocket + 移动端推送触发 |
| **生产 INFO 告警** | `is_sandbox=false`，置信度 < 0.8 | 仅写数据库，不推送 |
| **连续帧确认** | 告警触发后，第 1、2 帧未连续 | 不触发推送（防误报） |
| **时窗归并** | 同类告警 5 分钟内重复出现 | 第 2 条及以后静默，不重复推送 |
| **is_sandbox 篡改防御** | 消息中 `is_sandbox` 字段被恶意设为 `false` | 服务拒绝并记录安全审计日志 |
| **notification 防御性校验** | notification-service 收到 `is_sandbox=true` 消息 | 直接丢弃，不推送，记录拦截日志 |

#### 2.2.3 低代码编排器后端（覆盖率 ≥ 85%）

| 用例 | 验收标准 |
|---|---|
| 工作流 JSON 序列化/反序列化 | 7 种节点类型 Round-trip 无数据丢失 |
| 节点连接合法性校验 | 孤立节点、回路、缺少视频源节点时返回 `1xxx` 错误 |
| 配置热更新下发 | 修改后配置版本号 +1，旧版本可回滚 |
| 乐观锁并发保护 | 两个请求同时 PUT 同一场景，后者返回错误码 `2001` |

### 2.3 API 接口测试用例设计

#### 2.3.1 设计原则

- **契约优先**：所有接口响应必须严格符合统一格式 `{code, message, data, trace_id}`
- **异常路径覆盖**：每个接口至少设计 3 条异常用例（参数缺失、权限不足、资源不存在）
- **幂等性验证**：`PUT`/`DELETE` 接口重复调用结果一致
- **错误码精确性**：返回码与接口规范文档完全一致

#### 2.3.2 关键接口用例列表

**鉴权接口（3 个）**

| 接口 | 正常用例 | 异常用例 |
|---|---|---|
| `POST /api/v1/auth/login` | 正确账密返回 JWT | 错误密码返回 `1001`；账号不存在返回 `3001`；5次失败锁定 |
| `POST /api/v1/auth/refresh` | 有效 refresh_token 返回新 access_token | 过期 token 返回 `4001`；伪造 token 返回 `4002` |
| `POST /api/v1/auth/logout` | 正常登出，token 加入黑名单 | 重复 logout 返回 `200`（幂等） |

**场景管理接口（8 个）**

| 接口 | 关键用例 |
|---|---|
| `POST /api/v1/scenes` | 成功创建；重复 scene_id 返回 `2001`；VIEWER 角色返回 `4003` |
| `PUT /api/v1/scenes/{id}` | 乐观锁 version 不匹配返回 `2001`；并发写场景冲突 |
| `POST /api/v1/scenes/{id}/enable` | 未绑定摄像头返回 `2003`；未配置模型返回 `2003` |
| `GET /api/v1/scenes/{id}/workflow` | 返回 workflowJson 含 nodes + edges |
| `PUT /api/v1/scenes/{id}/workflow` | 空 nodes 数组合法；保存后版本号 +1 |

**Sandbox 接口（5 个）**

| 接口 | 关键用例 |
|---|---|
| `POST /api/v1/sandbox/sessions` | 创建会话，topic 自动设为 `tianjing.frame.sandbox` |
| `POST /api/v1/sandbox/sessions/{id}/stop` | 停止后 status=STOPPED；重复停止幂等 |
| `GET /api/v1/sandbox/sessions/{id}/report` | 报告包含三项门禁状态（accuracy/resource/latency） |
| `POST /api/v1/sandbox/sessions/{id}/promote` | 门禁未通过时返回 `2007`；验证时长 < 48h 返回 `2005` |
| 四眼原则验证 | 提交人自审返回 `4004`；不同 REVIEWER 可通过 |

**模型管理接口（5 个）**

| 接口 | 关键用例 |
|---|---|
| `POST /api/v1/models/{id}/approve` | 自审返回 `4004`；approved=false 写 rejected_reason |
| `POST /api/v1/models/{id}/promote` | status 非 approved 时返回 `2003`；触发蓝绿切换 |
| 状态机流转 | STAGING→REVIEWING→PRODUCTION，不得跳步 |

**告警管理接口（4 个）**

| 接口 | 关键用例 |
|---|---|
| `POST /api/v1/alarms/{id}/feedback` | 重复反馈返回 `2008`（幂等检查） |
| Sandbox 告警查询 | `is_sandbox=true` 的告警只出现在实验室视图，不在生产告警列表 |

#### 2.3.3 通用校验项（每个接口）

```
□ 响应包含 trace_id 字段
□ 分页接口响应包含 total, page, size, items
□ Content-Type: application/json
□ 无 Authorization Header → 401
□ 角色不足 → 403
□ 路径参数不存在 → 3xxx 错误码
□ 必填参数缺失 → 1001
□ SQL 注入字符串不导致 500
□ 超大请求体（>10MB）返回 413
```

### 2.4 集成测试用例设计

#### 2.4.1 推理全链路集成测试

**测试场景：烧结看火单帧录像回放端到端**

```

- notification-service 消费队列无 is_sandbox=true 的消息

```

## 3. 测试环境规划

### 3.1 环境分层

| 环境 | 用途 | 数据来源 | GPU |
|---|---|---|---|
| **dev**（本地） | 开发自测、单元测试 | Mock / 录像文件 | 无（CPU 推理） |

## 8. 缺陷分级与处理流程

### 8.1 缺陷等级定义

| 等级 | 定义 | 对应 CLAUDE.md | 处理时限 |
|---|---|---|---|
| **P0 - 安全红线** | Sandbox 推理结果触发生产告警推送；is_sandbox 被篡改；数据库越权访问 | P0 禁止行为 | 立即停止，当日修复，上报负责人 |
| **P1 - 严重** | 核心业务流程阻断（无法登录/告警/推理）；四眼原则失效；数据丢失 | P1 架构红线 | 24h 内修复，不得带缺陷上线 |
| **P2 - 主要** | 功能不符合需求、性能超阈值（P99 超限 50% 以上）、错误码不符合规范 | P2 工程规范 | 72h 内修复，Sprint 内解决 |
| **P3 - 次要** | 界面样式问题、非核心路径的错误提示文字、文档错误 | — | 下个 Sprint 修复或接受 |
| **P4 - 建议** | 用户体验优化、性能优化建议 | — | Backlog 排期 |

### 8.2 缺陷处理流程

```
发现缺陷
  → 记录（标题/复现步骤/截图/日志/trace_id/严重等级）
    → P0：立即停止当前测试，通知技术负责人 → 紧急修复 → 回归验证
    → P1：创建 Bug 工单，优先级 Highest → 24h 内修复 → Sprint 内验收
    → P2：创建 Bug 工单，优先级 High → 72h 内修复 → 回归测试
    → P3/P4：记录至 Backlog → 下次迭代评估
```

## 10. 风险与规避措施

| 风险 | 概率 | 影响 | 规避措施 |
|---|---|---|---|
| GPU 到货延期，性能测试无法执行 | 高 | 高 | Sprint 3-4 用 CPU 模式验证逻辑正确性，GPU 到货后补充性能回归 |
| IIoT 平台联调接口变更 | 中 | 高 | Sprint 3 用 Mock Broker（EMQX 本地），Sprint 4 再接甲方真实环境 |
| 测试录像数据不足（场景覆盖不全） | 中 | 中 | 预留 Sprint 3 第 1 周用于数据收集和脱敏，建立测试数据集管理规范 |
| Sandbox 拦截器回归风险 | 低 | P0 | 每个 Sprint 的 CI 流水线中强制运行 Sandbox 隔离专项测试（不可跳过） |
| 告警误报率高（影响甲方信任） | 中 | 高 | Sprint 3 专设防误报机制测试（连续帧确认 + 时窗归并），误报率目标 < 20% |
| 四眼原则被绕过（审核合规风险） | 低 | 高 | 服务端强校验 + 单元测试 100% 覆盖 + 每次发布前回归验证 |
| 测试环境与生产环境配置不一致 | 中 | 中 | 使用 Helm Values 分层管理，test/staging/prod 同一套 Chart，仅参数不同 |
| 并发告警风暴（大量告警积压 Kafka） | 低 | 中 | 压测中验证时窗归并机制，明确 Kafka 消费者组扩容阈值 |
