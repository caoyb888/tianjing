# 模型训练使用说明与完成度评估

> **编写日期**：2026-04-04
> **适用范围**：天柱·天镜平台 — 从训练数据到生产模型的完整流程说明及现状评估

---

## 一、完整流程总览

```
[1] 准备训练数据集           → 管理后台：训练数据集页面
       ↓
[2] 提交训练作业             → 管理后台：训练作业页面 → 提交训练作业
       ↓
[3] 训练脚本执行（训练环境）  → K8s Job + PyTorch/MMDetection
       ↓
[4] MLflow 记录实验结果      → mlflow_run_id / mlflow_model_uri 写入 train_job 表
       ↓
[5] 注册为模型版本            → 管理后台：模型版本管理 → 提交新模型
       ↓
[6] 提交审核                 → 模型详情页 → "提交审核"
       ↓
[7] 四眼审批                 → 由另一名 MODEL_REVIEWER 审批通过
       ↓
[8] 蓝绿上线                 → 状态变为 PRODUCTION，旧版本自动 DEPRECATED
       ↓
[9] 推理服务加载新模型        → model-loader 读取 mlflow_model_uri 热加载
```

---

## 二、各步骤详细操作说明

### 步骤 1：准备训练数据集

**页面路径**：管理后台 → 训练管理 → 训练数据集（`/training/datasets`）

数据集在 `tianjing_train` 库的 `dataset` 表中注册，关键字段：

| 字段 | 说明 |
|---|---|
| `dataset_code` | 数据集编号，如 `DS-SINTER-001` |
| `dataset_name` | 数据集名称 |
| `factory_code` | 厂部代码（pellet / sintering / steel / section / strip） |
| `total_samples` | 图像总数 |
| `positive_samples` | 正样本数（有缺陷） |
| `minio_prefix` | MinIO 存储路径前缀，格式：`tianjing-datasets/{factory}/{dataset_code}/` |

数据集版本（如 `v1.0`、`v2.0`）通过 `dataset_version_tag` 表关联，提交训练作业时需指定具体版本 ID。

---

### 步骤 2：提交训练作业

**页面路径**：管理后台 → 训练管理 → 训练作业（`/training/jobs`）→ 提交训练作业

**所需角色**：ADMIN

**API 接口**：`POST /api/v1/training/jobs`（由 `drift-monitor-service` 提供）

**提交参数**：

```json
{
  "plugin_id": "ATOM-DETECT-YOLO-V1",
  "dataset_version_id": "DS-SINTER-001-v2.0",
  "gpu_count": 2,
  "train_config_json": {
    "epochs": 100,
    "lr": 0.001,
    "batch_size": 16,
    "img_size": 640
  }
}
```

提交成功后，`train_job` 表中创建一条状态为 `PENDING` 的记录，作业ID格式如 `TJ-ATOM-DETECT-YOLO-V1-1743500000000`。

---

### 步骤 3：训练脚本执行（训练环境，自动）

提交后，训练平台（K8s Job）自动在 `training` 命名空间中启动训练容器：

```python
# training/train_jobs/ 中的训练脚本（示例）
import mlflow
import torch

with mlflow.start_run() as run:
    mlflow.log_params({"epochs": 100, "lr": 0.001, "batch_size": 16})

    for epoch in range(epochs):
        # 训练循环 ...
        mlflow.log_metrics({"map50": map50, "loss": loss}, step=epoch)

    # 保存模型至 MLflow
    mlflow.pytorch.log_model(model, "model",
        registered_model_name="ATOM-DETECT-YOLO-V1")

    print(f"MLflow Run ID: {run.info.run_id}")
    # 训练完成后，通过内部 API 回写 train_job 状态和 mlflow_run_id
```

训练过程中，`train_job.status` 依次变化：`PENDING → RUNNING → COMPLETED`

训练完成后，`train_job` 表记录最终指标：
- `best_map50`：最佳 mAP@50
- `best_map50_95`：最佳 mAP@50-95
- `mlflow_run_id`：MLflow 实验运行 ID（训练溯源关键字段）

---

### 步骤 4：注册为模型版本（手动）

训练完成后，在管理后台手动将训练产物注册为平台可管理的模型版本。

**页面路径**：管理后台 → 模型管理（`/models`）→ 提交新模型

**API 接口**：`POST /api/v1/models`

**所需角色**：ADMIN 或 MODEL_REVIEWER

**提交参数**（对应 `ModelRegisterRequest`）：

```json
{
  "plugin_id": "ATOM-DETECT-YOLO-V1",
  "mlflow_run_id": "abc123def456",
  "model_artifact_url": "models:/ATOM-DETECT-YOLO-V1/1",
  "training_job_id": "TJ-ATOM-DETECT-YOLO-V1-1743500000000"
}
```

注册成功后，`model_version` 表中创建一条版本记录，初始状态为 `STAGING`，版本ID格式如 `MV-XXXXXXXX`。

---

### 步骤 5：提交审核

**页面路径**：管理后台 → 模型管理 → 点击版本ID → 详情页 → "提交审核"

**条件**：模型当前状态为 `STAGING` 或 `SANDBOX_VALIDATING`

点击后状态变为 `REVIEWING`，系统记录 `submitted_by`（提交人用户名）。

---

### 步骤 6：四眼审批

**页面路径**：同上详情页 → "审批通过" / "审批拒绝"

**所需角色**：ADMIN 或 MODEL_REVIEWER

**四眼原则**：审核人不能与提交人为同一账号。若同一用户尝试审批，按钮置灰并提示，API 返回错误码 `4004`。

- **审批通过**：状态 `REVIEWING → PRODUCTION`，同一插件的旧 PRODUCTION 版本自动变为 `DEPRECATED`（蓝绿切换）
- **审批拒绝**：状态 `REVIEWING → STAGING`（回退，可重新修改后再提交）

---

### 步骤 7：推理服务热加载（自动）

模型上线后，`inference/model-loader/` 服务监听状态变化，通过 `mlflow_model_uri` 从 MLflow Artifacts 拉取模型文件并热替换到推理服务，无需重启。

### ❌ 尚未完成（需 Sprint 后续实现）

#### 1. 实际训练执行引擎（最核心缺口）

**位置**：`DriftController.triggerRetrain()` 第 47 行注释：`// 实际训练 K8s Job 创建逻辑（Sprint 3 完善）`

当前状态：提交训练作业仅在数据库中创建 PENDING 记录，**没有任何机制触发实际训练**。K8s Job 的创建、训练容器的调度、状态回写均未实现。

`training/` 目录下所有子目录（`data_pipeline/`、`train_jobs/`、`eval/`、`export/`）均为空（仅有 `.gitkeep`）。

**影响**：训练作业提交后永远停在 `PENDING` 状态，不会产生任何训练结果。

---

#### 2. 训练完成后自动注册模型版本的衔接

当前需要人工从 MLflow 查询 `run_id`，再手动到"提交新模型"表单填写。训练作业详情页没有"注册为模型版本"的一键按钮，`train_job_id` 也未在模型注册表单中展示。

---

#### 3. 前端提交表单字段与后端不匹配（现有 Bug）

`TrainingJobListView.vue` 提交表单发送：
```json
{ "sceneId": "...", "datasetCode": "...", "epochs": 100 }
```

但后端 `TrainJobRequest` 期望：
```json
{ "plugin_id": "...", "dataset_version_id": "...", "train_config_json": {...}, "gpu_count": 1 }
```

字段名完全不一致，提交作业功能当前**实际不可用**。

---

#### 4. 数据集上传功能

目前只能查看数据集，无法通过 UI 上传图像或添加新数据集记录。数据集须通过 SQL 直接写入 `tianjing_train` 库，或通过外部工具上传到 MinIO。

---

#### 5. Sandbox 验证阶段

`SANDBOX_VALIDATING` 状态在模型状态机中存在，但从训练完成 → Sandbox 验证 → 转正的完整链路未串通。`compare-dashboard-service` 和 `sandbox-infer-service` 均为空（`.gitkeep`）。

---

#### 6. 训练数据库种子数据

`tianjing_train` 库缺少 `scripts/seed_drift_monitor.sql`，本地开发环境无法直接看到测试数据集和历史训练作业。

---

## 四、当前可实际操作的最短路径

在训练引擎未完成前，若需将一个已有模型纳入平台管理，可走以下手动流程：

```
1. 手动将模型文件上传到 MinIO：tianjing-models-staging/{plugin_id}/
2. 在 MLflow 中手动注册模型版本，获取 mlflow_run_id 和 model_uri
3. 管理后台 → 模型管理 → 提交新模型（填写 plugin_id、mlflow_run_id、model_artifact_url）
4. 模型详情页 → 提交审核
5. 由另一名 MODEL_REVIEWER 登录 → 审批通过
6. 模型自动上线（PRODUCTION），推理服务热加载
```
