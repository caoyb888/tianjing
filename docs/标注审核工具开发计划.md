# 标注审核工具（Annotation Review）开发计划

> **版本**：V1.0 · **日期**：2026-04-06  
> **适用范围**：`history-replay-service` 后端 + `admin-web` 前端  
> **前置条件**：仿真任务状态为 `COMPLETED`，`result_json` 已写入帧推理结果

---

## 一、问题背景与目标

### 1.1 现有链路的缺陷

当前"仿真 → 导出训练集"全链路无人工介入：

```
推理引擎输出（draft 标注）
  → DatasetExportService.doExport()
  → 按 confThreshold 过滤
  → COCO JSON + 帧图像 zip
  → tianjing-datasets/{versionId}/dataset.zip
  → 训练作业
```

**核心问题**：模型的漏检帧直接变成负样本、误报帧直接变成含错误框的正样本，导致训练数据质量无法保证。

### 1.2 目标功能

在导出前插入人工审核环节：

```
推理引擎输出（draft 标注）
  → 标注审核工具（人工确认 / 修改 bbox / 拒绝帧）
  → 审核后标注（存入 simulation_frame_review 表）
  → DatasetExportService 优先读取审核标注
  → 高质量 COCO 数据集 → 训练作业
```

### 1.3 不做的事情

- 不做跨任务标注对比
- 不做语义分割 / 关键点标注（仅 bbox）
- 不做多人协同审核（单操作员）
- 不做标注版本历史回溯

## 二、数据模型

### 2.1 新增表：`simulation_frame_review`

**Flyway 迁移文件**：`deploy/flyway/prod/V10__annotation_review.sql`

```sql
CREATE TABLE simulation_frame_review (
    id                          BIGSERIAL      PRIMARY KEY,
    task_id                     VARCHAR(64)    NOT NULL,   -- 关联 simulation_task.task_id
    frame_id                    VARCHAR(64)    NOT NULL,   -- 帧编号，如 frame_000001
    frame_url                   VARCHAR(512)   NOT NULL,   -- MinIO 帧图像地址（来自 result_json）
    frame_index                 INTEGER        NOT NULL,   -- 帧序号（0-based，用于分页排序）

    -- 推理原始结果（初始化时从 result_json 复制，只读参考）
    original_detections_json    TEXT           NOT NULL DEFAULT '[]',

    -- 人工校正结果（NULL = 未修改使用原始结果；'[]' = 明确标注为无目标负样本）
    corrected_detections_json   TEXT           NULL,

    review_status   VARCHAR(16)    NOT NULL DEFAULT 'PENDING',
    -- PENDING   = 未审核
    -- APPROVED  = 通过（含 is_modified=false 原样通过 / is_modified=true 修改后通过）
    -- REJECTED  = 排除（此帧不纳入训练集）
    -- SKIPPED   = 暂跳过（等同 PENDING，导出时按配置决定是否包含）

    is_modified     BOOLEAN        NOT NULL DEFAULT false, -- 人工是否修改过检测框
    reviewed_by     VARCHAR(64)    NULL,
    reviewed_at     TIMESTAMPTZ    NULL,
    created_at      TIMESTAMPTZ    NOT NULL DEFAULT NOW(),

    CONSTRAINT uq_frame_review UNIQUE (task_id, frame_id),
    CONSTRAINT ck_review_status CHECK (review_status IN ('PENDING','APPROVED','REJECTED','SKIPPED'))
);

CREATE INDEX idx_frame_review_task_id     ON simulation_frame_review (task_id);
CREATE INDEX idx_frame_review_task_status ON simulation_frame_review (task_id, review_status);
```

**`corrected_detections_json` 数组元素结构**（与 `result_json.frames[].detections` 保持一致）：

```json
[
  {
    "classId": 0,
    "className": "壁条脱落",
    "confidence": 1.0,
    "bbox": { "x1": 120.0, "y1": 80.0, "x2": 340.0, "y2": 260.0 }
  }
]
```

> 人工新增的框 confidence 固定写 `1.0`，以区分模型输出的置信度。

### 2.2 对现有表的影响

无结构变更。`simulation_task` 新增一列用于记录审核进度（可选，通过 `ALTER TABLE` 追加）：

-- 含义：0 = 未开始审核；1~99 = 审核中（已审帧/总帧数 × 100）；100 = 全部审核完成

## 三、后端 API 设计

所有接口挂载于 `SimulationController`，路径前缀 `/api/v1/simulations/{task_id}/review`。

### 3.1 接口清单

| 方法 | 路径 | 说明 |
|---|---|---|
| `POST` | `/{task_id}/review/init` | 初始化审核：从 `result_json` 批量创建 `simulation_frame_review` 记录 |
| `GET` | `/{task_id}/review/stats` | 审核进度统计（总帧数 / 已通过 / 已拒绝 / 待审） |
| `GET` | `/{task_id}/review/frames` | 分页获取帧列表（带审核状态，用于缩略图导航） |
| `GET` | `/{task_id}/review/frames/{frame_id}` | 获取单帧详情（含原始标注 + 校正标注 + 图像URL） |
| `PUT` | `/{task_id}/review/frames/{frame_id}` | 保存单帧校正结果（标注框 + 审核状态） |
| `POST` | `/{task_id}/review/bulk-approve` | 批量通过（支持按 frame_ids 列表或"全部未改动帧"） |

### 3.2 接口详细说明

#### `POST /review/init`

- 幂等：若已存在记录则跳过，不覆盖
- 逻辑：解析 `simulation_task.result_json`，为每个 `frame` 创建一条 `simulation_frame_review` 记录，`original_detections_json` 填入推理 detections
- 响应：`{ "total": 16, "already_existed": 0, "created": 16 }`

#### `GET /review/frames?page=1&size=20&status=PENDING`

```json
{
  "total": 16,
  "page": 1,
  "size": 20,
  "items": [
    {
      "frameId": "frame_000001",
      "frameUrl": "http://minio:9000/tianjing-sim-temp/simulation/SIM-xxx/frames/frame_000001.jpg",
      "frameIndex": 0,
      "reviewStatus": "PENDING",
      "isModified": false,
      "detectionCount": 2,          // 原始检测框数量
      "correctedCount": null         // 校正后数量（未审核时 null）
    }
  ]
}
```

#### `PUT /review/frames/{frame_id}`

请求体：
```json
{
  "review_status": "APPROVED",
  "corrected_detections": [
    {
      "class_id": 0,
      "class_name": "壁条脱落",
      "confidence": 1.0,
      "bbox": { "x1": 120.0, "y1": 80.0, "x2": 340.0, "y2": 260.0 }
    }
  ]
}
```

- `corrected_detections` 为 `null` → `is_modified = false`，导出时使用原始推理结果
- `corrected_detections` 为空数组 `[]` → `is_modified = true`，标注为负样本
- `corrected_detections` 有内容 → `is_modified = true`，使用校正后的框

#### `POST /review/bulk-approve`

请求体：
```json
{
  "mode": "all_unmodified"   // 或 "by_ids"
  // "frame_ids": ["frame_000001", "frame_000002"]   // mode=by_ids 时使用
}
```

`all_unmodified` 模式：将所有 `is_modified = false` 且 `review_status = PENDING` 的帧批量设为 `APPROVED`，适用于"置信度够高、大部分帧不需要改"的快速审核场景。

### 3.3 `DatasetExportService` 修改点

位置：`DatasetExportService.doExport()` 第 130 行附近，过滤逻辑之前插入：

```java
// 查询该任务的审核记录（key: frameId → review record）
Map<String, SimulationFrameReview> reviewMap = loadReviewMap(task.getTaskId());
boolean hasReview = !reviewMap.isEmpty();
```

过滤循环中，每帧的 detections 来源改为：

```java
for (FrameResult frame : frames) {
    SimulationFrameReview review = reviewMap.get(frame.frameId());

    // 有审核记录且状态为 REJECTED → 跳过此帧
    if (review != null && "REJECTED".equals(review.getReviewStatus())) continue;

    List<Detection> detections;
    if (review != null && Boolean.TRUE.equals(review.getIsModified())
            && review.getCorrectedDetectionsJson() != null) {
        // 使用人工校正结果（confidence=1.0，不再过滤）
        detections = parseDetections(review.getCorrectedDetectionsJson());
    } else {
        // 使用原始推理结果，仍按 confThreshold 过滤
        detections = filterByConf(frame.detections(), confThreshold);
    }
    // ... 后续生成 COCO 标注的逻辑不变
}
```

同时，`DatasetExportRequest` 新增可选字段：

```java
/** 导出模式：ALL（全部帧）/ REVIEWED_ONLY（仅已审核通过帧）*/
@JsonProperty("export_mode")
String exportMode();  // 默认 "ALL"
```

`REVIEWED_ONLY` 模式下，无审核记录或状态非 `APPROVED` 的帧全部跳过。

## 六、关键技术风险与应对

### 6.1 MinIO 跨域访问帧图像

**风险**：`AnnotationCanvas` 直接加载 MinIO 图像 URL 时，浏览器可能因 CORS 拒绝访问，导致 `<img>` 能显示但 Canvas drawImage 失败（如需 canvas 截图时）。

**应对**：本项目纯 SVG 覆盖层方案中，标注绘制完全在 SVG 层，不需要 `canvas.drawImage()`，CORS 问题不影响功能。若未来需要截图导出预览图，通过 history-replay-service 代理下载帧图像（新增 `GET /simulations/{task_id}/frames/{frame_id}/image` 代理端点）。

### 6.2 帧数量级与分页

**风险**：5 fps × 10 分钟视频 = 3000 帧，一次加载全部缩略图会卡顿。

**应对**：
- 缩略图列表虚拟滚动（Element Plus `el-virtual-list` 或手写虚拟列表）
- 缩略图接口 `GET /review/frames` 固定 `size=50`，左侧栏滚动到底部时自动加载下一页
- 缩略图使用懒加载 `loading="lazy"`

### 6.3 SVG 坐标缩放精度

**风险**：用户操作的是缩放后的 SVG 坐标（如 800×450 的显示区域），需精确映射回 1920×1080 原图坐标。

**应对**：在 `useCanvasCoords.ts` 中统一维护 `scaleX = imgNaturalWidth / displayWidth`，所有存储操作在 `mouseup` 时执行 `Math.round(displayCoord × scale)` 转换。图像加载完成时（`onImageLoad`）重新计算 scale。

### 6.4 `DatasetExportService` 数据源切换兼容性

**风险**：修改导出逻辑可能影响现有无审核记录的任务（V9 之前的历史仿真任务）。

**应对**：
- `reviewMap` 为空时（`loadReviewMap` 返回空 Map）完全走原有逻辑，保持向后兼容
- `export_mode = "ALL"`（默认值）时不区分审核状态，对未审核帧仍按 confThreshold 处理
- 仅 `export_mode = "REVIEWED_ONLY"` 时才跳过非 APPROVED 帧

## 七、数据流全景（实现后）

```
仿真任务 COMPLETED
    │
    ▼
POST /review/init
    │  解析 result_json.frames
    │  为每帧创建 simulation_frame_review (PENDING)
    ▼
AnnotationReviewView 逐帧审核
    │  帧图像：MinIO tianjing-sim-temp/.../frames/
    │  原始框：original_detections_json
    │  操作：增 / 删 / 改 bbox
    │
    ├── PUT /review/frames/{id}  { status=APPROVED, corrected_detections=[...] }
    ├── PUT /review/frames/{id}  { status=REJECTED }
    └── POST /review/bulk-approve { mode=all_unmodified }
    ▼
POST /export-dataset { export_mode="REVIEWED_ONLY" }
    │
    ▼
DatasetExportService.doExport()
    │  ┌─ frame 有 APPROVED 记录 + is_modified=true  → 用 corrected_detections
    │  ├─ frame 有 APPROVED 记录 + is_modified=false → 用 original_detections（confThreshold 过滤）
    │  └─ frame 有 REJECTED 记录                    → 跳过
    ▼
COCO JSON + 帧图像 zip
    → tianjing-datasets/{versionId}/dataset.zip
    ▼
训练作业（高质量标注）
```
