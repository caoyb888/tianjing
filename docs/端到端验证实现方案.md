> **版本**：V1.0 · **日期**：2026-04-04
> **目标**：以三台物理服务器 + 录像回放 + 公网推理的方式，跑通从"录像帧 → Kafka → 推理 → 告警"的完整链路，支撑 Sprint 3"烧结看火单场景全功能端到端验证"（S3-02～S3-07）

**数据完整流转路径：**
```
边缘录像
  → recording-replay-service（Server 2/3）
  → Kafka: tianjing.frame.production（Server 1，外部端口 9094）
  → route-dispatch-service（Server 1）
  → cloud-inference-proxy（Server 1，port 8092）
  → 公网推理 API
  → result-aggregate-service（Server 1）
  → alarm-judge-service（Server 1，Sandbox 拦截器校验）
  → TDengine（告警记录持久化）
  → notification-service（WebSocket / MQTT 推送）
  → Admin Web 实时告警大屏（Server 1）
```

## 三、首要实现方案：最快半天跑通全链路

> **目标**：在半天内完成从"录像帧→Kafka→推理→告警"的完整链路验证，不依赖 GPU、不依赖 K8s、不依赖现场摄像头。

### 阶段一：Server 1 启动（约 1.5 小时）

**Step 1：启动中间件**

```bash
# Server 1
cd /home/xintong/tianjing/deploy/docker

# 配置环境变量（首次运行需要）
cp .env.example .env
# 编辑 .env，重点设置：
#   KAFKA_EXTERNAL_HOST=10.30.10.49   ← Server 1 的内网 IP，边缘服务器需要
#   POSTGRES_PASSWORD=xxx
#   MINIO_ROOT_USER=minioadmin
#   MINIO_ROOT_PASSWORD=xxx

docker compose up -d

# 等待所有服务 healthy（约 2 分钟）
docker compose ps

# 验证：Kafka、PostgreSQL、MinIO、Redis 全部 healthy 后继续
```

**Step 2：启动平台后台服务**

```bash
# Server 1（按照规范，必须通过 start-backend.sh 启动）
cd /home/xintong/tianjing
bash scripts/start-backend.sh

# 验证关键服务存活
curl http://localhost:8081/actuator/health  # scene-config-service
curl http://localhost:8083/actuator/health  # alarm-rule-service
curl http://localhost:8089/actuator/health  # drift-monitor-service
```

**Step 3：启动云端推理代理**

```bash
# Server 1
cd /home/xintong/tianjing/inference/cloud-inference-proxy
pip install fastapi uvicorn opencv-python-headless onnxruntime httpx structlog

# 方案A（推荐先用）：ONNX CPU 本地推理，不需要公网 API Key
export INFER_BACKEND=onnx_cpu
export ONNX_MODEL_PATH=/path/to/yolov8n.onnx  # 从 ultralytics 下载
export CONF_THRESHOLD=0.25

# 方案B：对接公网API
# export INFER_BACKEND=cloud_api
# export CLOUD_API_ENDPOINT=https://your-cloud-api.com/detect
# export CLOUD_API_KEY=your-api-key

nohup python src/main.py > /tmp/cloud-proxy.log 2>&1 &
```

> **无 YOLOv8n ONNX 模型时**：代理服务会返回空检测结果（`detections: []`），链路仍然完整可跑通，只是不产生告警。下载命令：
> ```bash

> python -c "from ultralytics import YOLO; YOLO('yolov8n.pt').export(format='onnx')"

> ```

## 六、公网推理 API 对接说明

云端推理代理当前支持的两种模式：

### 模式 A：ONNX CPU 本地推理（优先验证链路）

```bash
export INFER_BACKEND=onnx_cpu
export ONNX_MODEL_PATH=/models/yolov8n.onnx
```

- 零成本，无需 API Key
- 使用 YOLOv8n 通用预训练权重（未针对工业场景微调，精度低）
- **Sprint 3 目标是验证链路，不验证精度，此模式完全满足**

### 模式 B：公网 API（真实推理效果）

```bash
export INFER_BACKEND=cloud_api
export CLOUD_API_ENDPOINT=https://your-cloud-api.com/detect
export CLOUD_API_KEY=your-api-key
```

云端推理代理对不同公网 API 的响应格式有一个适配层（`_infer_cloud_api` 函数），不同云平台的响应字段不同，需要根据实际 API 文档调整解析逻辑（`cloud-inference-proxy/src/main.py` 第 175 行附近）。

### 模式切换

两种模式通过环境变量切换，**重启服务即生效，不需要修改任何代码**。这是 V2.0 架构的设计目标：GPU 到货后同理，只需在管理后台将场景绑定的算法插件从 `CLOUD-PROXY-V1` 切换为本地 GPU 推理插件，无需改代码。

## 八、与模型训练的衔接

本方案验证推理链路时**不依赖真实训练模型**（使用 YOLOv8n 预训练权重或公网 API），因此：

- Sprint 3 S3-13（训练全流程）和 S3-02（推理链路）可以**并行推进**，互不阻塞
- 推理链路（本文档）在本方案验证通过后，模型训练优化（见《模型训练优化计划.md》T-01～T-05）完成后，两条线在 S3-13 汇合
- Sprint 5 GPU 到货后，只需将边缘服务器部署本地 GPU 推理容器，`cloud-inference-proxy` 退出历史舞台
