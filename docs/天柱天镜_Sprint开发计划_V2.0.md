## 策略变更说明（V1.0 → V2.0）

本次调整由项目负责人审核后确定，核心思路从**"硬件与软件并行推进"**调整为**"软件先行、单场景验证、硬件解耦"**，具体体现在四个方面：

| 调整点 | V1.0 策略 | V2.0 策略 | 影响 |
|--------|----------|----------|------|
| **① GPU 服务器** | Sprint 0 即上架，推理任务强依赖 | GPU 到货前使用**云端推理代理**替代，GPU 相关任务移至硬件到位后专项 Sprint | 推理任务不阻塞软件开发；M1-M3 里程碑不再依赖 GPU 到货 |
| **② 开发顺序** | 场景上线与软件同步推进，互相依赖 | **后端 API 全部建设完毕 → 前端全部页面完成 → 再进行场景验证**，顺序解耦 | 软件质量提升，前后端联调完整，不会出现"为赶场景忽略平台功能"的情况 |
| **③ 测试方式** | 依赖现场摄像头实时流 | **以实验室录像为基础**，通过录像回放服务注入 Kafka 管道，复现完整数据流 | 开发团队在办公室即可完成完整功能验证，不依赖现场环境 |
| **④ 上线节奏** | 多场景并行推进 | **先以"烧结看火"单场景验证系统全部功能闭环，再横向扩展其他场景** | 降低风险；系统整体逻辑在第一个场景中完整打通后，扩展其他场景边际成本极低 |

### 两个关键使能组件（V2.0 新增）

V2.0 能够做到软件与硬件解耦，依赖以下两个新增技术组件：

**① 云端推理代理服务（Cloud Inference Proxy）**
```
标准插件接口（Image In → JSON Out）
         ↓
   云端推理代理容器
         ↓
   公共云视觉 API（如阿里云视觉智能 / Baidu EasyDL / CPU ONNX Runtime）
```
代理服务对平台上层完全透明，行为与本地 GPU 推理容器完全相同，仅性能（延迟/精度）存在差异。GPU 到货后，只需将场景绑定的算法插件切换至本地 GPU 推理容器，无需修改平台任何代码。

**② 录像回放服务（Recording Replay Service）**
```
MinIO 存储的录像文件（已有烧结看火视频）
         ↓
  录像回放服务（可调速：0.5x / 1x / 2x）
         ↓
  Kafka Topic: tianjing.frames.prod（与实时摄像头推送完全相同格式）
         ↓
  路由引擎 → 推理服务 → 告警引擎（完整管道不变）
```
平台下游对帧来源无感知，录像回放等价于真实摄像头接入。

## 全局 Definition of Done

1. **代码**：合并至 `develop`，PR 经 ≥1 人 Code Review 通过
2. **测试**：Service 层单元测试覆盖率 ≥80%；关键链路集成测试通过
3. **文档**：接口变更同步 API 规范；DB 变更同步数据库设计文档
4. **安全**：无 SQL 注入风险，接口有权限注解，日志无敏感信息
5. **日志**：关键操作有含 `trace_id` 的 INFO 日志
6. **验收**：Story 所有验收项 ✅ 通过
