# 系统冷启动情况说明

> **适用场景**：全新部署的天柱·天镜平台，`model_version` 表为空，MinIO 中尚无任何训练产出的 ONNX 模型文件。
> **编写日期**：2026-04-07

---

## 一、问题背景

完整的模型使用链路如下：

```
① 训练作业（yolo-trainer）
   输入: tianjing-datasets/{versionId}/dataset.zip
   输出: tianjing-models-staging/HEAD-SINTER-GRATE-V1/{jobId}/best.onnx
              ↓
② 人工审核 + CI/CD 发布（MODEL_REVIEWER 四眼原则）
   tianjing-models-staging → tianjing-models-prod
              ↓
③ 更新推理代理配置
   ONNX_MODEL_PATH = /models/HEAD-SINTER-GRATE-V1/best.onnx
   重启 cloud-inference-proxy
              ↓
④ 新建仿真任务
   → cloud-inference-proxy 用新模型推理
   → 帧数/告警帧不再为零
```

**先有鸡还是先有蛋**：没有训练好的模型 → 无法跑出有效检测 → 无法产出带标注的数据集 → 训练作业无法运行。

---

## 二、关键结论：仿真任务不依赖已训练模型

查阅代码可知，系统在模型缺失时**不会崩溃，而是优雅降级**：

| 组件 | 无模型时的行为 |
|---|---|
| `cloud-inference-proxy` `_load_onnx_session()` | 文件不存在 → 记录 warning → 返回 `None` |
| `cloud-inference-proxy` `/infer` | session 为 `None` → 返回空检测列表 `[]` |
| `InferenceClient.warmup()` | 预热失败 → 记录 warn → **不中止仿真** |
| `InferenceClient.infer()` | 推理异常 → 返回 `List.of()` → **不抛出异常** |
| `SimulationExecutor` | 所有帧正常抽取、上传 MinIO → 写入 `result_json` |

**最终结果**：仿真任务照常完成。
- `total_frames` = 实际抽帧数（**非零**）
- `matched_alarms` = 0（无检测，因为没有模型）
- 帧图像正常上传至 MinIO `tianjing-sim-temp`
- `result_json` 中每帧的 `detections: []`

数据集导出（Phase 3）同样可以正常运行，导出的 `dataset.zip` 包含所有帧图像，标注文件为空。

---

## 三、三条冷启动路径

### 路径比较

| 路径 | 操作复杂度 | 帧数 | 告警帧 | 适用场景 |
|---|---|---|---|---|
| **① 空检测启动** | 最低，无需额外操作 | 正常（非零） | 0 | 从零开始，人工从头打标签 |
| **② 种子模型**（推荐） | 低，上传一个通用 ONNX | 正常（非零） | 有（通用 COCO 类） | 希望管线完整端到端跑通并验证 |
| **③ 云 API 后端** | 低，修改两个环境变量 | 正常（非零） | 有（云 API 结果） | 已有云视觉平台账号 |

---

### 路径①：空检测启动（零依赖，立刻可用）

**不需要任何额外操作**，直接按如下流程冷启动：

```
1. 上传视频文件到仿真库
2. 新建仿真任务（选任意 pluginId，如 CLOUD-PROXY-V1）
3. 仿真完成：total_frames=N（非零），matched_alarms=0
4. 提交数据集导出 → 下载 dataset.zip（N 张帧图，全空标注）
5. 人工用标注工具（LabelImg / Label Studio）对帧图逐一打标签
6. 上传带标注的 zip 到 MinIO tianjing-datasets/{versionId}/dataset.zip
7. 提交训练作业（yolo-trainer）
8. 产出第一个 best.onnx → 走正常审核流程上线
```

**优点**：无需准备任何模型或外部服务，系统现有代码即可支持。
**缺点**：检测帧为 0，人工标注工作量大，需要标注人员对工业场景有足够经验。

---

### 路径②：种子模型（推荐，管线可端到端验证）

使用通用预训练的 YOLOv8n ONNX（在 COCO 80 类上训练，~6MB）作为引导模型。
系统 `cloud-inference-proxy` 默认已配置 `ONNX_MODEL_PATH=/models/yolov8n.onnx`，只需放入文件即可。

#### 步骤 1：下载通用种子模型并上传至 MinIO

```bash
# 在运维机或 CI/CD 环境执行
wget -O yolov8n.onnx https://github.com/ultralytics/assets/releases/download/v8.2.0/yolov8n.onnx

# 上传至 MinIO 生产模型桶
mc cp yolov8n.onnx minio/tianjing-models-prod/bootstrap/yolov8n.onnx
```

#### 步骤 2：在 model_version 表注入引导记录

执行 `scripts/seed_model_version.sql`（**仅系统首次部署时执行一次**）：

```sql
-- SECURITY: 引导豁免 — 仅系统冷启动使用，跳过四眼原则审核
-- 依据：新系统无历史模型，此为管线可用的最低前提条件
-- 上线首个正式版本后，本记录状态将自动变更为 DEPRECATED（蓝绿切换机制）
INSERT INTO model_version (
    version_id, plugin_id, status, model_path, export_format,
    mlflow_run_id, train_job_id,
    created_by, updated_by, created_at, updated_at
) VALUES (
    'MV-BOOTSTRAP-0001',
    'CLOUD-PROXY-V1',
    'PRODUCTION',
    'tianjing-models-prod/bootstrap/yolov8n.onnx',
    'ONNX',
    NULL, NULL,
    'system', 'system', NOW(), NOW()
) ON CONFLICT (version_id) DO NOTHING;
```

#### 步骤 3：挂载模型文件至推理容器

在 `cloud-inference-proxy` 的 Docker Compose / Helm values 中配置 volume 挂载：

```yaml
# docker-compose（本地开发）
volumes:
  - ./bootstrap-models/yolov8n.onnx:/models/yolov8n.onnx:ro

environment:
  ONNX_MODEL_PATH: /models/yolov8n.onnx
  INFER_BACKEND: onnx_cpu
```

重启 `cloud-inference-proxy` 后，`/health` 接口的 `onnx_model_loaded` 字段将变为 `true`。

#### 步骤 4：跑仿真并收集第一批数据

```
新建仿真任务 → 仿真完成（帧数=N，告警帧>0，有通用类检测结果）
→ 人工复核：保留正确框，删除误检框，补充漏检框
→ 提交数据集导出
→ 提交训练作业
→ 产出第一个域内 best.onnx → 走正常审核流程
```

**优点**：管线从头到尾均有真实数据流，便于早期验证系统集成。通用模型的检测结果（即使类别名不对）可辅助标注人员快速定位目标区域。
**注意**：种子模型仅用于冷启动期间数据采集，不作为任何生产告警依据。

---

### 路径③：云 API 后端（无需本地模型）

若已有云视觉平台账号（阿里云视觉智能 / 百度 EasyDL），可直接对接：

修改 `cloud-inference-proxy` 环境变量（通过 Nacos 配置中心下发或修改 start-backend.sh 对应配置）：

```bash
INFER_BACKEND=cloud_api
CLOUD_API_ENDPOINT=https://your-cloud-vision-api/detect
CLOUD_API_KEY=your-api-key
```

重启 `cloud-inference-proxy` 即可，无需本地模型文件。

**优点**：最快速，无需准备 ONNX 文件，云 API 结果通常质量较高。
**缺点**：每次仿真推理都调用付费 API；帧图像上传至云端，需评估数据合规要求（工业生产图像可能涉密）。

## 四、推荐操作顺序

```
阶段 0（冷启动验证）
  └─ 用路径①或②跑一次仿真
     → 确认 total_frames > 0
     → 确认帧图像出现在 MinIO tianjing-sim-temp
     → 确认数据集导出生成 dataset.zip

阶段 1（第一次训练）
  └─ 人工标注导出帧 → 上传 dataset.zip
     → 提交训练作业（yolo-trainer）
     → 等待产出 best.onnx（存入 tianjing-models-staging）

阶段 2（正式审核上线）
  └─ MODEL_REVIEWER 四眼原则审核
     → CI/CD 推送至 tianjing-models-prod
     → 更新 ONNX_MODEL_PATH → 重启 cloud-inference-proxy
     → 后续仿真使用域内模型，告警帧数量显著提升

阶段 3（持续迭代）
  └─ 生产推理 → 人工复核 → 漂移监测 → 触发重训练
     → 完全进入正常的模型迭代闭环
```

## 五、常见问题

**Q：仿真任务状态变为 FAILED 怎么办？**
A：检查 `cloud-inference-proxy` 是否正常启动（`GET /health` 是否返回 200）。推理失败不会导致仿真 FAILED，视频下载失败或 FFmpeg 解码失败才会。查看 `history-replay-service` 日志中 `sim-exec-*` 虚拟线程的异常栈。

**Q：路径②中种子模型的检测类别名称是 `class_0`、`class_1` 这样的，正常吗？**
A：正常。`cloud-inference-proxy` 中通用 YOLOv8n 的类别映射尚未配置（`main.py:148` 中 `class_name=f"class_{class_id}"`）。这不影响数据集导出，标注人员在复核时会手工修正标注框并指定正确类别。

**Q：如何判断模型是否已成功挂载并加载？**
A：调用推理代理健康检查接口：
```bash
curl http://localhost:8092/health
# 返回 "onnx_model_loaded": true 即为已加载
```

**Q：种子模型的 `model_version` 记录会影响生产告警吗？**
A：不会。`model_version` 表仅用于模型版本管理和 UI 展示，生产推理服务实际加载的模型由 `ONNX_MODEL_PATH` 环境变量决定。仿真推理全程携带 `is_sandbox=true`，`alarm-judge-service` 的 Sandbox 拦截器会拦截所有仿真结果，保证不触发任何生产告警。
